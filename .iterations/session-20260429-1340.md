# Cycle session note — 2026-04-29 13:40

**Cycle 20 of the /loop 25m orchestration (cron-fired).** Picked queue item (a) — the manual ImageData ellipse rasterisation perf optim that's been on the queue since cycle 7 and was the top remaining engineering item per PROJECT-STATE. **Implemented in `src/style/Pointillism.js`** behind a `manualRaster: false` opt and `--manual-raster` CLI flag. The path: pre-fetch ImageData once, in the stroke loop compute rotated-ellipse bounding box → per-pixel rotation + hit-test → alpha-blend in place into the buffer, putImageData back to canvas after all strokes. Hypothesis was ~25% speedup by skipping `ctx.fillStyle` parsing and `ctx.beginPath/ellipse/fill` setup overhead per stroke. **Result: hypothesis falsified — manual raster is ~16% SLOWER** (14.6 s avg vs canvas-API's 12.6 s on the same storm-seascape + mountain-twilight benchmark). The reason: node-canvas's native ellipse rasterizer is heavily optimized in C++; pure-JS per-pixel rotation + hit-test loses to it. The Canvas2D overhead I theorized about is dominated by the actual per-pixel rasterisation work, which native code wins. Visual output is identical at production density+opacity (the AA loss is imperceptible because strokes stack densely enough to mask it). **Decision: keep `--manual-raster` flag as documented opt-in for pixel-exact output, but canvas-API stays the default. Negative result locked in code comments and PROJECT-STATE.md so future cycles don't repeat the experiment.** **Cycle 20 deliverable:** working manualRaster code path, documented finding, updated PROJECT-STATE.md queue (item 1 struck through with explanation).
