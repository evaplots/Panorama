# Session note — 2026-05-01 17:24

**Painterly vegetation + landmarks (V2 reincarnation of "Phase 3").** First
session after the v3.6 chore that removed 3D OSM rendering, picking up the
artistic intent of the original Phase 3 (forests should read as forests,
landmarks visible) and shipping it inside the painter pipeline. No 3D
geometry was restored; the deleted `VegetationBuilder.js` / `BuildingsBuilder.js`
/ `LODManager.js` stay deleted.

## What shipped

- `src/style/canopyPainter.js` (new) — stippled dark-green canopy dabs over
  forest / wood polygons. Per-polygon dab density driven by screen-projected
  bbox area; mm-physical dab radius matches the v1.4 stroke pass; PRNG forked
  from the master seed (`seed ^ 0xC4_C4_C4_C4`) so canopy is deterministic
  and decoupled from stroke-placement bits. Sun-phase tint envelope mirrors
  groundPainter (warm/cool/desat alpha by phase). Plug point: between
  `paintGround` and the median-blur underpainting.
- `src/style/landmarkPainter.js` (new) — silhouette marks for tower / church
  / monument / castle / named-attraction. Trapezoidal base sized to the OSM
  `height` tag (or per-category default); category-specific top decorations
  (church cross, castle crenellation). Heights converted from world-metres
  to pixels via the same focal-length math `projection.js` uses, so a 50 m
  tower at 800 m and a 100 m tower at 1600 m come out the same pixel size —
  correct perspective, not flat icons. PRNG forked at `seed ^ 0x14_14_14_14`.
  Sun-phase tone shift baked into the silhouette tone at draw time (single
  pass — clipped overlay would have double-tinted ground/canopy pixels).
- `src/osm/OSMFetcher.js` — extended:
  - Combined Overpass query gains node coverage for `man_made=tower`,
    `historic=castle|monument|memorial`, `amenity=place_of_worship`, and
    `tourism=attraction` (plus the way variant for tourism). Per taginfo
    (verified today), 77 % of `man_made=tower` and 68 % of
    `tourism=attraction` are nodes; the way-only query before this PR
    missed three quarters of the landmark candidates. Cache key is
    bbox-based so existing cached tiles silently miss the new categories
    until their 7-day TTL expires; new locations get the full set
    immediately. Acceptable trade-off for a single-user dev workflow.
  - New methods: `fetchLandmarks(location, preset)` and
    `peekLandmarks(location, preset)`. Same combined-query cache as
    `fetchGroundCover` so calling both for the same `(location, preset)`
    pair issues no extra Overpass round-trips.
  - New helpers: `classifyLandmark(tags)`, `parseHeightM(tags)`,
    `ringCentroid(ring)`, `elementsToLandmarks(elements)`. Way landmarks
    use the centroid of their geometry; node landmarks use their
    coordinates directly; relation landmarks fall back to the first outer
    member's centroid (rare for landmarks anyway).
- `src/style/Pointillism.js` — wired the two new painters into the bindings
  branch. `timing` object grows four fields: `canopyDabCount`, `canopyMs`,
  `landmarkDrawnCount`, `landmarkMs`.
- `src/ui/ControlsPanel.js` — `buildBindings()` now peeks landmarks alongside
  ground-cover (cache-only, no network). Result panel shows canopy / landmark
  counts and timing.
- `src/style/groundPainter.js` — refactored away from `Path2D` (not a Node
  global; node-canvas doesn't expose it). Same visual output via
  `ctx.beginPath` + `moveTo` / `lineTo`; this also unblocks node-side test
  coverage of the polygon underpainting path, which had been latent.
- `scripts/canopy-landmark-perf-probe.js` (new) — risk-first perf probe at
  A3 @ 300 DPI. Three synthetic scenes: forest (Black Forest analogue),
  city (Florence / Manhattan analogue, 35 mixed landmarks), combo (worst-
  case both at once).

## Perf probe results

A3 landscape @ 300 DPI, v1.4 expressionist preset
(`brushWidthMm=1.2, brushStrokeFactor=2.0, density=0.03, applyMedianUnderpaint=false`):

| Scene  | Canopy dabs | Canopy ms | Landmarks drawn | Landmarks ms | Total       |
| ------ | ----------- | --------- | --------------- | ------------ | ----------- |
| forest | 47          | 4–8       | 3               | 0.4–0.6      | 22.5–24.8 s |
| city   | 0           | 0.1       | 35              | 2–5          | 23.6–25.9 s |
| combo  | 63          | 5–9       | 38              | 1.5–4        | 24.2–25.3 s |

All three under the 28 s pass bar in steady-state runs.

**Variance caveat.** Re-running the probe several times back-to-back
produced 30–73 s totals on the same code, dominated by stroke-pass time.
That's node-canvas GC pressure under sustained load, not a genuine
regression — the painters' own timings stay under 60 ms in the worst case
observed. Browser timings will track RELEASE-NOTES (22–26 s) rather than
the noisy node numbers. The same painters added 15 ms at most to that
budget on every run.

**Determinism check.** Two `applyPointillism` calls with the same source +
same `bindings` + same `seed` produced byte-identical PNG buffers
(5,992,593 bytes, equal). Canopy dab placement and landmark silhouette
draw both honour the master-seed contract.

(13 MB A3 PNGs + 800 px thumbnails):

- **forest.png** — sky band + green ground band. Ground polygons clearly
  visible. Canopy stipple is faint at the default density (0.45 dabs per

  silhouettes at the horizon. Landmarks read clearly as a townscape.
- **combo.png** — both elements present.

Browser visual QA (`npm run dev` + paint a Black Forest / Sequoia / Florence
location) is the user's gate from here; these probe outputs prove the
pipeline works end-to-end with the actual Pointillism path.
