# Session note — 2026-05-01 22:34

**Atmospheric depth — Phase 5 polish (haze + bloom + grain/grading).**
Branch `feature/atmospheric-depth`, draft PR follows. Three closely related
post-passes that share an architectural slot (end-of-underpainting whole-
canvas passes) and tune together. Combined PR per brief — haze without
bloom looks sterile, bloom without haze looks pasted-on, grain without
either reads as noise.

The live underpainting preview shipped two sessions ago is the
development instrument. All three passes drop in after the existing
`paintGround → waterPainter → canopyPainter → landmarkPainter →
median blur` sequence, before the canvas is returned to the stroke pass.

## Decisions made up front (the brief invited these as questions)

- **Depth proxy: vertical-screen-position with horizon-aware curve.**
  Per-pixel altitude-from-horizon is derived from screen Y relative to
  the horizon line the projection module already exposes (`projector.horizonY()`).
  Pixels above the horizon get zero haze (sky); pixels just below the
  horizon get strong haze (distant terrain at the horizon line); pixels
  far below get tapered haze (foreground). A real depth buffer would
  require touching the terrain step (forbidden by the brief) and the
  marginal accuracy gain doesn't justify the architectural cost given
  the painter consumes a 2D snapshot. Surface this trade-off in the PR.

  This depth proxy also naturally satisfies "haze should obey scene
  scale": an alpine valley with a far horizon line dominates more of
  the canvas → more pixels in the heavy-haze band → strong recession.
  An urban courtyard fills the canvas with foreground polygons → most
  pixels are well below the horizon → barely any haze. Probe 1 verifies
  this empirically.

- **Grain: uniform with mid-tone weighting, NOT brightness-modulated.**
  Brightness-modulated grain (uniform on darks, attenuated on lights)
  reads as "paper texture" but is also harder to control across phases:
  at night the whole image is dark and the grain becomes overpowering.
  Uniform amplitude with a small mid-tone weighting (peak at L≈0.5,
  falloff toward black/white) keeps grain visible across the dynamic
  range without ever overpowering shadows or highlights — the painted-
  on-textured-paper feel the brief wants. Documented as a Decision Log
  entry; future PR can revisit if curation surfaces a need for fully
  uniform or fully brightness-keyed grain.

- **Bloom radius: 18 mm at full strength, scaled with `bloomStrength`.**
  Wider than a 3D engine bloom (which is typically 5–10 mm screen
  equivalent) because the painter reads as a painting, and a painted
  sun's halo is a generous patch — think Turner's Norham Castle, not a
  digital lens-flare. Soft radial gradient, additive blend, warm-tinted
  by sun phase via the SUN_PHASE_TINT envelope.

## Probe results

`scripts/atmospheric-perf-probe.js` — three probes, all PASS.

### 1. Haze depth correctness (alpine vs courtyard)

Alpine vista (1500 m observer, looking south across a 12 km valley) vs
urban courtyard (street level, 30 m residential polygon filling the
foreground). Same haze strength (0.5) for both.

| Scene     | Horizon-band sat | Foreground-band sat | Delta (fg − horizon) | Haze ms |
| --------- | ---------------: | ------------------: | -------------------: | ------: |
| alpine    |            0.403 |               0.502 |               +0.099 |     7.4 |
| courtyard |            0.538 |               0.187 |               −0.352 |     2.5 |
| **Differential** | | | **+0.451** | |

Alpine shows clear horizon-band desaturation relative to the foreground
(positive delta — the haze pass hits the distant band heavier than the
foreground). Courtyard shows a NEGATIVE delta — the horizon band is
*more* saturated than the foreground band, because there's no real
horizon for the haze to act on. The differential of +0.45 between the
two scenes confirms the depth proxy is scene-scale aware.

### 2. Bloom horizon gate (4 times of day)

Coastal observer with elevationDeg=10 (level-ish camera so all four
sun altitudes project predictably). Sun azimuth fixed at 180°
(directly ahead), altitude varies.

| Time     | Altitude | Phase         | Expected | Observed | Match |
| -------- | -------: | ------------- | -------- | -------- | ----- |
| 6am      |       4° | goldenHour    | YES      | YES      | ✓     |
| noon     |      25° | day           | YES      | YES      | ✓     |
| sunset   |       2° | sunset        | YES      | YES      | ✓     |
| midnight |     −25° | night         | NO       | NO       | ✓     |

Bloom fires only when the sun is above horizon AND its projected screen
position falls within the canvas — exactly the brief's gate.

PNGs saved to `.iterations/2026-05-01-atmospheric-depth/bloom-{6am,noon,sunset,midnight}.png`.

### 3. Perf at A3 @ 300 DPI

Coastal extent scene, expressionist preset (brushWidthMm=1.2,
brushStrokeFactor=2.0, density=0.03), atmospherics enabled at default
slider values.

| Metric                | Value     | Bar             | Verdict |
| --------------------- | --------: | --------------- | ------- |
| Haze ms               |   ~85–150 | < 80 ms target  | ⚠ over  |
| Bloom ms              |       ~2  | < 80 ms target  | ★ PASS  |
| Grain ms              |  ~620–650 | < 80 ms target  | ⚠ over  |
| Atmospherics total    |     ~720  | —               | —       |
| **TOTAL A3 render**   |   24.3 s  | **< 28 s**      | **PASS** |

Total A3 render passes the 28 s user-tolerance bar with ~3.7 s headroom.
The atmospherics overhead is ~720 ms (3 % of total).

**Per-pass overruns are surfaced per the brief's escalation path.** The
80 ms-per-pass target is fundamentally tight for any per-pixel JS write
on a 17.4 MP canvas:

- A getImageData + putImageData round-trip alone is ~70 ms in node-canvas
  before any per-pixel work — that already eats the budget.
- Haze (~85 ms typical) is essentially at the bar; cell-based downsampling
  isn't applicable (haze is a single-pass alpha blend across the
  below-horizon strip).
- Grain (~620 ms) IS already using the brief's recommended downsampling
  (cell-based at GRAIN_CELL_MM=0.18 mm — at 300 DPI, ~2.1 px cells, so
  the noise buffer is ~1/4 the canvas pixel count). The dominant cost is
  the per-pixel read-modify-write to ImageData, not the noise lookup.

Three faster alternatives were tried and discarded:
1. **Native `drawImage` scale-up** of a small noise canvas with `overlay`
   composite at A3 → 686 ms. node-canvas's `overlay` composite at 17 MP is
   slower than per-pixel JS.
2. **`createPattern` + `fillRect`** with `overlay` at A3 → 584 ms. Same
   reason — `overlay` at 17 MP is the bottleneck, regardless of source.
3. **Per-pixel JS without mid-tone weighting** → 622 ms. The mid-tone
   weight wasn't the cost; the per-pixel write to ImageData is.

If grain perf becomes a curation pain point, the fix is to do grain in a
WebGL fragment shader rather than per-pixel JS. That's a meaningful
architectural change — out of scope for this PR; the painter has been
canvas2D-only since Phase 2.5.
